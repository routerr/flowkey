# flowkey Deep Review — 2026-04-11

This document captures a deep analysis of the `flowkey` codebase as of 2026-04-11,
identifying functional gaps, stability risks, and cross-platform rough edges.
It is paired with [`work-backlog-2026-04-11.md`](./work-backlog-2026-04-11.md),
which translates the findings below into an AI-agent-friendly execution plan.

---

## 1. Scope Recap

`flowkey` is a CLI-first Rust workspace (plus a Tauri management GUI) for a
LAN-only keyboard/mouse sharing tool targeting macOS and Windows. V1 requires:

- same local network, both machines logged in
- explicit hotkey switching
- keyboard and mouse only
- CLI operation with GUI companion

Deferred: clipboard sync, file transfer, screen-edge switching, login-screen
control, public internet support.

## 2. Current State — Strengths

The following paths are in good shape and verified end-to-end between macOS and
Windows (see `docs/cross-platform-test-report.md`):

- Rust workspace with clean crate boundaries.
- Ed25519 node identity, signed pairing tokens, mutual authenticated TCP
  handshake, heartbeat session pump.
- `flowkey-net` framed protocol with **Bincode** binary serialization,
  microsecond capture-time timestamps, bounded channels.
- Native capture and injection backends for macOS (`CGEventTap` / `rdev::grab`)
  and Windows (`rdev::grab` + `enigo`), with loopback suppression and
  activation-chord suppression.
- **macOS Unix Domain Socket IPC** for sub-10ms `flky switch` response.
- FRP (reachability probing) for multi-interface zero-config switching.
- mDNS discovery, `flky doctor`, `flky setup` interactive wizard.
- Tauri GUI scaffolding with React/TS dashboard, system tray, pairing radar and
  6-digit SAS confirmation flow.
- Packaging scripts for `.dmg`, `.zip`, and `tar.gz` release bundles.

## 4. Uncommitted Changes — What They Tell Us

Five files are currently modified but not committed. They are clearly
debugging/hardening attempts aimed at G3 (GUI lifecycle) and are worth
committing with minor adjustments:

- `flowkey-gui/src/main.rs` — adds a panic hook that writes to
  `~/Library/Logs/flowkey-panic.log` on macOS. Good signal.
- `flowkey-input/src/capture.rs` — replaces `lock().expect(..)` with
  `lock().ok()?` and drops the event on poison. **Risk**: silently losing input
  events is a subtle regression; we should at minimum `tracing::warn!` before
  dropping and then re-initialize the loopback.
- `flowkey-input/src/native_injector.rs` — same pattern applied to
  `record_loopback` paths. Lower risk because missing a loopback record only
  weakens self-filter, not injection itself.
- `flowkey-platform-macos/src/capture.rs` — poisoned-mutex fallbacks in both
  `grab` and `listen` callbacks. Same warning applies.

These edits strongly suggest a recent panic in the loopback suppressor mutex
path, likely triggered by the capture thread crashing while another thread was
mid-`lock()`. The right long-term fix is G5 (decouple the big mutex) and G8
(capture thread watchdog); the short-term fix (ship these diffs + log) is still
the correct interim move.

## 5. Action Plan Overview

The work breaks naturally into three phases, detailed in the companion
backlog:

- **Phase A — Stabilize Windows & GUI (≈1 week)**: Named Pipe IPC parity on
  Windows, `app.manifest` with `uiAccess`, supervised daemon lifecycle inside
  the GUI, pairing peer address persistence. Goal: daily use without silent
  failures.
- **Phase B — Reliability Hardening (≈1–2 weeks, roadmap Milestone 9)**: split
  `DaemonRuntime` state, forced-release recovery with tests, capture thread
  watchdog, mouse move coalescing, reconnect chaos tests.
- **Phase C — UX & Release Polish (≈1 week)**: macOS permission deep-link,
  native installers (WiX + notarization), auto-start, remote-control toggle in
  GUI, refactor `native_injector.rs` into platform crates, repo hygiene.

## 6. Cross-Platform Smoothness Targets

At the end of Phases A+B, `flowkey` should meet:

- **Latency**: `flky switch` < 10ms on both platforms; end-to-end input
  latency (LAN) < 15ms p95.
- **Stability**: 10-second Wi-Fi drop recovers without user intervention; no
  stuck modifiers after disconnect; `cargo test` exercises sticky-key and
  reconnect scenarios.
- **Windows UIPI**: daemon can control an elevated Task Manager window (with
  signed binary and `uiAccess=true`).
- **macOS permissions**: `flky doctor` and GUI can deep-link into System
  Settings and recheck after grant.
- **GUI resilience**: daemon panic restarts automatically and surfaces the
  event to the user.

See the backlog for acceptance criteria per task.
